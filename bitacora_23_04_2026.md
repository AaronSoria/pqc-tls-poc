# Bitácora — PQC TLS Benchmark (Go)
**Fecha:** 23 de abril de 2026
**Proyecto:** `pqc-tls-poc` — Migración del benchmark Python a Go

---

## Contexto

Los resultados del benchmark Python tenían un problema fundamental: el cliente AWS PQ usaba `subprocess` para lanzar `s2nc` en cada iteración. Esto generaba ~88ms de overhead por proceso (inicialización de AWS-LC), contaminando completamente las mediciones. La solución fue migrar todo el benchmark a Go con las siguientes estrategias por stack:

| Stack | Cliente anterior | Cliente nuevo |
|---|---|---|
| Classical TLS | subprocess Python + ssl | `crypto/tls` Go nativo — cero subprocess |
| OQS PQC | subprocess openssl s_client | subprocess openssl s_client (sin alternativa) |
| AWS PQ | subprocess s2nc (~88ms overhead) | CGO directo contra `libs2n` + AWS-LC |

---

## Problema 1: `undefined symbol: EVP_aead_aes_128_gcm_tls13`

### Síntoma
```
/usr/local/lib/libs2n.so.1: undefined symbol: EVP_aead_aes_128_gcm_tls13
benchmark exited with code 127
```

### Causa
El `Dockerfile.benchmark` compilaba AWS-LC como shared library (`-DBUILD_SHARED_LIBS=ON`) y ejecutaba `ninja install`, que copiaba `libcrypto.so` a `/usr/local/lib`. En runtime, el linker encontraba primero la `libcrypto` del sistema (OpenSSL estándar de Ubuntu), que no tiene `EVP_aead_aes_128_gcm_tls13` — ese símbolo es exclusivo de AWS-LC.

### Solución
Compilar AWS-LC y s2n-tls como **librerías estáticas** (`-DBUILD_SHARED_LIBS=OFF`). El binario Go linka `libcrypto.a` y `libs2n.a` directamente en tiempo de compilación — no hay búsqueda en runtime y no hay conflicto con el sistema.

Adicionalmente, el OpenSSL de OQS se instaló en `/opt/oqs-openssl` (prefijo aislado) en vez de `/usr/local`, evitando cualquier colisión futura.

```dockerfile
# AWS-LC — solo estático
cmake -GNinja -DCMAKE_BUILD_TYPE=Release -DBUILD_SHARED_LIBS=OFF ..

# s2n-tls — estático contra AWS-LC estático
cmake -GNinja -DBUILD_SHARED_LIBS=OFF \
    -Dcrypto_LIBRARY=/opt/aws-lc/build/crypto/libcrypto.a ..

# OQS OpenSSL — prefijo aislado
./Configure --prefix=/opt/oqs-openssl ...
```

## Problema 2: "Broken pipe" en aws-pq-server

### Síntoma
```
aws-pq-server | Unexpected IO error during shutdown: Broken pipe
aws-pq-server | Error closing connection: 'underlying I/O operation failed'
```

### Causa
El cliente CGO cerraba el socket TCP directamente sin hacer el TLS shutdown graceful (`s2n_shutdown`). El servidor detectaba el cierre abrupto como pipe roto.

### Solución
Agregar `s2n_shutdown(conn, &blocked)` antes del bloque `cleanup` en el cliente C embebido en `awspq.go`. Error cosmético que no afectaba mediciones, pero se corrigió para logs limpios.

## Arquitectura final del benchmark Go

```
benchmark-go/
├── cmd/benchmark/main.go         — entrypoint, lee env vars, imprime tabla
└── internal/
    ├── stats/stats.go            — mean, p50, p95, min, max
    ├── classical/classical.go    — crypto/tls Go nativo, socket directo
    ├── oqs/oqs.go                — subprocess openssl s_client + oqsprovider
    └── awspq/awspq.go            — CGO: s2n_negotiate() en proceso, sin subprocess
```

### Decisiones de diseño

**Classical:** `crypto/tls` de Go con `InsecureSkipVerify: true`. Se mide exclusivamente el handshake (`tlsConn.Handshake()`). Sin subprocess, sin overhead de proceso.

**OQS:** Sigue siendo subprocess de `openssl s_client` porque no existen bindings Go para `oqs-provider`. El overhead es ~1-2ms medible y documentado.

**AWS PQ:** CGO contra `libs2n.a` y `libcrypto.a`. `s2n_init()` se llama una sola vez por proceso (via `sync.Once`). El warmup paga la inicialización de AWS-LC una única vez. Las iteraciones miden solo el handshake neto.

## Resultados finales

**Entorno:** Docker containers en red interna (`pqc-net`), Ubuntu 22.04, 20 iteraciones por stack.

```
Stack                        Latency
---------------------------------------------------------------------------------------------
Classical TLS (RSA-2048)       1.28 ms (mean) |   1.26 p50 |   1.46 p95 |   1.18 min |   1.54 max | ok=20 fail=0
OQS PQC (X25519MLKEM768)      19.22 ms (mean) |  19.13 p50 |  19.87 p95 |  18.74 min |  19.88 max | ok=20 fail=0
AWS PQ (s2n + AWS-LC)         17.99 ms (mean) |  17.59 p50 |  19.75 p95 |  17.14 min |  20.33 max | ok=20 fail=0
---------------------------------------------------------------------------------------------
```

### Análisis

**OQS ≈ AWS PQ (~18-19ms):** Ambos usan X25519MLKEM768. La diferencia de ~1.2ms es el overhead del subprocess de `openssl s_client` en OQS, exactamente lo esperado. Confirma que las mediciones son consistentes entre stacks.

**Classical vs PQ (~14x):** El costo computacional de ML-KEM-768 es aproximadamente 14x mayor que RSA-2048 en este entorno local. En producción con red real (latencia de red > 10ms), este ratio se reduce significativamente porque la latencia de red domina el tiempo total de handshake.
