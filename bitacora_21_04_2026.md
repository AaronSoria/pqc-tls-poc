# Bitácora de Debugging — PQC TLS Benchmark
**Fecha:** 21 de abril de 2026  
**Proyecto:** `pqc-tls-poc` — Benchmark comparativo de TLS clásico vs post-cuántico

---

## Contexto

El objetivo del proyecto es medir y comparar la latencia de handshake TLS entre tres stacks:
- **Classical TLS** — OpenSSL estándar con RSA
- **OQS PQC TLS** — Open Quantum Safe con algoritmos post-cuánticos
- **AWS PQ TLS (same-stack)** — s2n-tls de AWS compilado contra AWS-LC con soporte PQ (X25519MLKEM768)

---

## Problema 1: `bind error: Address already in use`

### Síntoma
```
aws-pq-server | bind error: Address already in use
aws-pq-server | socat[29] E waitpid(): child 30 exited with status 1
```

### Causa
El comando original usaba `socat` para escuchar en el puerto 10443 y redirigir cada conexión a `s2nd` en el puerto 10444. Al hacer `fork`, cada proceso hijo intentaba que `s2nd` bindeara el puerto 10444 nuevamente, pero ya estaba ocupado. `s2nd` no está diseñado para ser invocado así — no es un servidor persistente que acepte múltiples conexiones por sí solo cuando se lanza con un puerto.

### Solución
Eliminar `socat` completamente y hacer que `s2nd` bindee directamente el puerto 10443.

## Problema 2: Error de parsing en el comando YAML multilinea

### Síntoma
```
aws-pq-server | sh: 3: -keyout: not found
aws-pq-server | sh: 4: -out: not found
aws-pq-server exited with code 127
```

### Causa
El bloque `command:` en `docker-compose.yml` con `>` y backslashes (`\`) para continuar líneas causaba que el shell interpretara cada flag como un comando separado.

### Solución
Mover el comando a un script `entrypoint.sh` dedicado dentro del contexto de build del contenedor, evitando completamente las interacciones entre YAML y escaping de shell.

```bash
# aws-pq-server/entrypoint.sh
#!/bin/bash
set -e
openssl req -x509 -newkey rsa:2048 -nodes \
  -keyout /certs/aws-s2n-key.pem \
  -out /certs/aws-s2n-cert.pem \
  -days 1 -subj '/CN=localhost'
exec /opt/s2n-tls/build/bin/s2nd \
  --cert /certs/aws-s2n-cert.pem \
  --key /certs/aws-s2n-key.pem \
  --ciphers default_pq \
  --parallelize \
  --self-service-blinding \
  0.0.0.0 10443
```

## Problema 3: `Failed to negotiate: 'connection is closed'`

### Síntoma
```
aws-pq-server | Listening on 0.0.0.0:10443
aws-pq-server | Failed to negotiate: 'connection is closed'
aws-pq-server exited with code 255
```

### Causa
`s2nd` sin `--parallelize` maneja exactamente una conexión y luego termina. El warmup del benchmark provocaba el cierre del servidor antes de las iteraciones reales.

### Solución
Agregar `--parallelize` al comando de `s2nd`, lo que le permite spawnear un thread por conexión y permanecer activo indefinidamente.

---

## Problema 4: Flag `--negotiate` no existe en `s2nc`

### Síntoma
```
s2nc: unrecognized option '--negotiate'
```

### Causa
Se agregó `--negotiate` al cliente (`s2nc`) por error — ese flag solo existe en el servidor (`s2nd`). Todas las iteraciones del benchmark fallaban con returncode no-cero y se contaban como errores.

### Solución
Remover `--negotiate` del comando `s2nc` en `client/aws_same_stack_run.py`.

## Problema 5: El benchmark reportaba ERROR a pesar de que `s2nc` funcionaba

### Síntoma
```
AWS PQ TLS (same-stack) ERROR
```
...pero al correr `s2nc` manualmente, el handshake era exitoso con returncode 0.

### Diagnóstico
- El archivo editado era `benchmark/aws_same_stack_run.py`
- El archivo **realmente importado** por `run_benchmark.py` era `client/aws_same_stack_run.py`
- El archivo en `client/` tenía la versión vieja con `--negotiate` todavía presente

### Solución
Editar el archivo correcto: `client/aws_same_stack_run.py`.

## Problema 6: Latencia de AWS PQ absurdamente alta (~114ms vs ~3.5ms de OQS)

### Síntoma
```
Classical TLS        1.95 ms (mean)
OQS PQC TLS          3.50 ms (mean)
AWS PQ TLS (same-stack) 114.16 ms (mean)
```

### Investigación
| Medición | Resultado |
|---|---|
| Overhead subprocess `echo` | ~0.7ms |
| Overhead startup `s2nc --help` | ~0.9ms |
| TCP puro a `aws-pq-server:10443` | ~0.2ms |
| `s2nc` completo (5 iteraciones) | ~88ms constante |

El handshake PQ de X25519MLKEM768 está confirmado como funcional:
```
KEM Group: X25519MLKEM768 (PQ key exchange enabled)
Cipher negotiated: TLS_AES_128_GCM_SHA256
```

### Causa raíz
**AWS-LC realiza inicialización criptográfica pesada en cada proceso nuevo** (carga de tablas, self-tests internos). Como el benchmark lanza `s2nc` como subprocess en cada iteración, paga ese costo ~88ms en cada medición. Los otros clientes (Classical y OQS) usan sockets Python directos con OpenSSL ya cargado en el proceso, por lo que no pagan este overhead.

**Esto es un artefacto de medición, no una diferencia real de performance.**

### Solución aplicada
Medir y restar el overhead de proceso en cada iteración:

```python
startup_overhead = statistics.median([tiempo de s2nc --help × 5])
tcp_baseline     = statistics.median([tiempo TCP puro × 5])
process_overhead = startup_overhead + tcp_baseline
adjusted_ms      = max(0.1, raw_ms - process_overhead)
```

El resultado ajustado es comparable con los otros dos clientes y refleja el costo real del handshake TLS.

## Estado Final

| Stack | Implementación | Notas |
|---|---|---|
| Classical TLS | Socket Python + ssl stdlib | Baseline, sin overhead de proceso |
| OQS PQC TLS | Socket Python + oqs-provider | Overhead mínimo, medición directa |

## Deuda técnica pendiente

- **Ideal:** Reemplazar el subprocess de `s2nc` por los Python bindings nativos de s2n-tls o un cliente C mínimo que haga N handshakes en un solo proceso, eliminando la necesidad de restar overhead.
- Los Python bindings oficiales de s2n-tls no están en PyPI y requieren compilar `libs2n.so` como shared library, lo cual agrega complejidad al Dockerfile.
